Refactoring Code

Saurav Biswas | Apr 3, 2021

Refactoring is the process of changing a software system in such a way that it does not alter the external behavior of the code, yet improves its internal structure.

It is a disciplined way to clean up code that minimizes the chances of introducing bugs. In essence, when you refactor, you are improving the design of the code after it has been written.” — Martin Fowler (Father of code smell).

Let’s face it. We are all programmers. We are smart and intelligent and like to deliver results. But how many times do we care about cleaning up code smell?

Perfect Code Please?

We desire to write code that is free of all bugs. One that tests all possible edge cases. One that passes linter checks each time you run a build. Code that you start to fall in love with. Surely top-notch programmers must be writing such code all the time, right?

Wrong.

No code is perfect. Ever!

Perfect code doesn’t exist, just like sacred interfaces don’t.

Code is something that evolves over time. In most cases, a codebase turns into a behemoth.

Complicated codebases are extremely difficult to manage.

Hence, it becomes all the more important to write clean code. And refactor stuff in bits and pieces. Take an apartment for example. It is more useful to clean it in short stints rather than one long session.

Whenever you tinker with a specific part of a codebase, ensure you follow up on the aspects below.

1. Check-in cleaner code

“Leave the campground cleaner than you found it.”— Clean Code (by Robert Martin)

The boy scout rule states that you must leave the campground cleaner than you found it. Since code degrades and rots over time, it is good practice to check in code cleaner than what you checked out from version control.

When making a code change, identify areas of improvement in the surrounding code.

2. Meaningful names

Meaningful names in your code ensure that you don’t have to add comments. The clearer the name, the easier it is for someone to read it and infer what it is doing.

When you use a clearer and more meaningful naming convention, you are essentially saving time to write developer documentation.

3. Responsibilities

Let’s face it. The longer a method body, the harder it is to follow. Look for lines that end up working towards a common goal. If you find such chunks, extract them out into separate methods with proper names.

The next person who works on that part of the code will thank you profusely. Trust me!

A method should be touching upon a single requirement at a time. Following this design paradigm goes a long way in reducing code complexity.

4. Design patterns

Design patterns enable you to implement tried and tested solutions to problems, thus saving time and effort during the implementation stage of the software development lifecycle.

By using well-understood and documented solutions, the final product will have a much higher degree of comprehension. If the solution is easier to comprehend, then by extension, it will also be easier to maintain.

5. Add tests

I can’t but stress the importance of testing your code. Ensure that you have enough unit and integration tests. These not only help you identify potential pitfalls and edge cases but also provide an opportunity to refactor stuff.

I can probably keep ranting about the importance of test coverage, and perhaps that warrants its own post one day.

Even Google recommends writing unit tests!

Know When to Draw the Line

I would conclude by saying that although refactoring is important, it should not come at the expense of completing your task at hand.

Be intelligent in choosing what and how much to refactor and very soon, you will be reaping the rewards of your diligence.
